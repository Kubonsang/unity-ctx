# v0.4c Apply + Diff Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a narrow `v0.4c` slice that can load the `v0.4b` `scene patch --op place_prefab` plan format, summarize it with `scene diff`, and execute it through `scene apply` with dry-run-first semantics and `--write`, while keeping scope limited to the current append-only patch-plan contract.

**Architecture:** Treat the `v0.4b` patch plan as the single source of truth. First, add a loader/validator for persisted patch plans plus fixtures. Second, add an append-only scene apply core that can render the current patch ops into Unity YAML blocks, perform dry-run summaries, and write with backup + parse verification. Third, expose `scene diff` and `scene apply` through service and CLI without widening beyond the existing `place_prefab` patch-plan schema.

**Tech Stack:** Go 1.22+, standard library only, existing `internal/patch`, `internal/parser`, `internal/document`, `internal/app`, `internal/cli`, `internal/mutation`, `testing`, `encoding/json`, `os`, `path/filepath`, `strconv`, `strings`

## Scope Decision

This plan intentionally keeps `v0.4c` smaller than “full scene mutation”:

- include now: persisted patch-plan loading/validation, `scene diff`, `scene apply` dry-run, `scene apply --write`, `.bak` backup, parse-after-write verification, append-only materialization for current `place_prefab` plans, fixture-backed tests, docs
- exclude now: full prefab-instance materialization, Editor GUID lookup, nested prefab semantics, `move_object`, `remove_object`, rule-driven placement constraints, scan/editor integration

Why this slice:

- `v0.4b` already established a deterministic patch-plan contract.
- the next useful step is taking that plan as input and safely previewing or applying it.
- full prefab-instance fidelity is still a separate complexity cliff; this slice should prove the dry-run/write pipeline first.

## Command Contract For This Slice

New commands in this slice:

```bash
unity-ctx scene diff testdata/scenes/simple_scene.unity --patch testdata/patches/chair_place_ok.patch.json

unity-ctx scene apply testdata/scenes/simple_scene.unity --patch testdata/patches/chair_place_ok.patch.json
# DRY_RUN

unity-ctx scene apply testdata/scenes/simple_scene.unity --patch testdata/patches/chair_place_ok.patch.json --write
# WRITE
```

Why `--patch` instead of a second positional argument:

- it preserves the existing global CLI shape of one positional `<file>` plus flags
- it keeps parser and contract changes smaller than introducing command-specific second positional files

Expected output shapes:

```text
OK op=place_prefab patch=testdata/patches/chair_place_ok.patch.json append_ops=2 reserved_fileIDs=2002,2003
DRY_RUN patch=testdata/patches/chair_place_ok.patch.json op=place_prefab append_ops=2 changed=1 verified=1
WRITE backup=testdata/scenes/simple_scene.unity.bak patch=testdata/patches/chair_place_ok.patch.json op=place_prefab append_ops=2 changed=1 verified=1
ERROR apply requires --patch
ERROR patch plan version mismatch version=2
ERROR APPLY_VERIFY_FAILED expected_objects=2 actual_objects=1
```

Rules:

- `diff` and `apply` in this slice are implemented only for the `scene` namespace.
- both commands require `--patch`.
- `apply` defaults to dry-run; `--write` is required for filesystem mutation.
- `diff` does not accept `--write`.
- both commands support only the current `place_prefab` append-only patch-plan schema from `v0.4b`.
- `apply --write` must create `.bak`, write once, then re-parse the scene.
- on committed write failure or parse verification failure, return `ERROR`.
- the patch file must be deterministic JSON with a known version.

## Patch File Contract

The persisted patch file should be the JSON emitted by `scene patch --json` in `v0.4b`.

For `v0.4c`, add a loader around the existing shape rather than inventing a second patch schema.

Expected top-level shape:

```json
{
  "status": "OK",
  "namespace": "scene",
  "command": "patch",
  "file": "testdata/scenes/simple_scene.unity",
  "view": "compact",
  "body": "OK ...",
  "patch_plan": {
    "status": "OK",
    "prefab_path": "Assets/Prefabs/chair.prefab",
    "prefab_guid": "guid-chair",
    "position": [5, 0, 0],
    "overlap_ids": [],
    "reserved_file_ids": [2002, 2003],
    "appends": [
      {"op": "append", "class_id": 1, "file_id": 2002, "type_name": "GameObject"},
      {"op": "append", "class_id": 4, "file_id": 2003, "type_name": "Transform"}
    ]
  }
}
```

Validation rules:

- top-level `command == "patch"`
- top-level `namespace == "scene"`
- `patch_plan` must be present
- `patch_plan.status` must be one of `OK`, `WARN`, `UNKNOWN`
- `patch_plan.prefab_path` non-empty
- `patch_plan.position` length is exactly 3
- `patch_plan.reserved_file_ids` count matches `patch_plan.appends`
- only append ops are accepted in this slice
- only `GameObject` and `Transform` append intents are accepted in this slice
