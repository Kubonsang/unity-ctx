# v0.4 Bounds Check Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver the first coherent `v0.4` slice by adding a deterministic bounds-manifest format plus `scene check` footprint validation, while deliberately deferring scene write operations (`patch/apply/diff`) until the read-side geometry contract is stable.

**Architecture:** Keep Unity Editor dependence at the edges. Define a manifest schema that can be produced by future `scan --mode editor`, but implement the core footprint-check engine as pure Go over scene data + manifest data + explicit placement input. The CLI should validate a proposed prefab placement, return deterministic overlap findings, and avoid any scene mutation.

**Tech Stack:** Go 1.22+, standard library only, existing `internal/parser`, `internal/document`, `internal/app`, `internal/cli`, `testing`, `encoding/json`, `math`, `os`, `path/filepath`, `strconv`, `strings`

---

## Scope Decision

This plan intentionally slices `v0.4` into the smallest coherent read-first foundation:

- include now: bounds manifest schema, manifest loader/validator, `scene check`, deterministic overlap/clear reporting, fixture-backed tests, docs
- exclude now: actual Unity Editor scanning implementation, `patch place_prefab`, `apply`, `diff`, scene write paths, prefab insertion, GUID/fileID allocation

Why this slice:

- `scan --mode editor` is useful, but Unity Editor invocation should not become the thing that blocks unit-tested core logic.
- `patch/apply/diff` depends on stable placement semantics first.
- `check footprint` gives us the geometry contract that later mutation commands must satisfy.

That makes this a clean `v0.4a` foundation instead of a half-finished placement writer.

## Proposed Command Contract For This Slice

Only one new command ships in this slice:

```bash
unity-ctx scene check testdata/scenes/simple_scene.unity \
  --manifest testdata/manifests/simple_scene.bounds.json \
  --prefab testdata/prefabs/chair.prefab \
  --position 0,0,0
```

Optional extension in the same slice if the core lands cleanly:

```bash
unity-ctx scene check testdata/scenes/simple_scene.unity \
  --manifest testdata/manifests/simple_scene.bounds.json \
  --prefab testdata/prefabs/chair.prefab \
  --position 0,0,0 \
  --json
```

Expected output shapes:

```text
OK clear=1 overlaps=0 prefab=Chair scene=testdata/scenes/simple_scene.unity
WARN clear=0 overlaps=2 ids=2000,2001 prefab=Chair scene=testdata/scenes/simple_scene.unity
ERROR check requires --manifest
ERROR MANIFEST_INVALID reason=missing_bounds object_id=2000
```

Rules:

- read-only command only; no write flag
- deterministic ordering for overlap ids
- explicit `WARN` instead of probabilistic “fit score”
- no guessing when manifest data is incomplete; return `ERROR` / `UNKNOWN` / `WARN`
- manifest must be external JSON, not inferred heuristically from raw YAML

## Manifest Contract

Use a deterministic JSON file produced by future editor scan work.

Suggested shape:

```json
{
  "scene": "Assets/Scenes/Stage01.unity",
  "source": "editor",
  "version": 1,
  "objects": [
    {
      "fileID": 2000,
      "name": "Chair_01",
      "bounds": {
        "center": [0.0, 0.5, 0.0],
        "size": [1.0, 1.0, 1.0]
      }
    }
  ],
  "prefabs": [
    {
      "path": "Assets/Prefabs/Chair.prefab",
      "bounds": {
        "center": [0.0, 0.5, 0.0],
        "size": [1.0, 1.0, 1.0]
      }
    }
  ]
}
```

For this slice, the engine only needs axis-aligned bounds in world space. Rotation-aware or mesh-accurate checks are explicitly deferred.

- [ ] **Step 2: Implement manifest types and validation**

Suggested types:

```go
type Vec3 [3]float64

type AABB struct {
	Center Vec3 `json:"center"`
	Size   Vec3 `json:"size"`
}

type ObjectBounds struct {
	FileID int64  `json:"fileID"`
	Name   string `json:"name"`
	Bounds AABB   `json:"bounds"`
}

type PrefabBounds struct {
	Path   string `json:"path"`
	Bounds AABB   `json:"bounds"`
}

type Manifest struct {
	Scene   string         `json:"scene"`
	Source  string         `json:"source"`
	Version int            `json:"version"`
	Objects []ObjectBounds `json:"objects"`
	Prefabs []PrefabBounds `json:"prefabs"`
}
```

Validation rules:

- `version == 1`
- `scene` non-empty
- object `fileID > 0`
- no duplicate `fileID`
- no duplicate prefab `path`
- every bounds `size` dimension must be `> 0`


Rules:

## Acceptance Criteria

This slice is done when:

- manifest JSON contract is validated in pure Go
- `scene check` works without Unity Editor
- overlap ids are deterministic
- no mutation occurs anywhere in the slice
- CLI rejects write/mutation-only flags for `check`
- docs reflect the narrowed `v0.4a` scope
- `go test ./...` passes

## Deferred To Later v0.4 Slices

Not covered here:

- `scan --mode editor` actual editor integration
- prefab bounds extraction from Unity Editor
- `patch place_prefab`
- `apply --write`
- `diff`
- rotation-aware placement
- terrain/slope/navmesh semantics
- nested prefab impact

## Recommended Execution Mode

Use **Subagent-Driven** execution.

Reason:

- manifest contract, geometry engine, and service/CLI wiring are cleanly separable
- fixture work is isolated
- review loops will matter because geometry and CLI semantics are easy places to drift
