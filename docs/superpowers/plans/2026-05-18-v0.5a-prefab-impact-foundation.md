# v0.5a Prefab Impact Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a read-only `unity-ctx prefab impact` command that finds where a target prefab is referenced across project `.unity` and `.prefab` files and reports deterministic compact/JSON output with nested-prefab depth warnings.

**Architecture:** Keep this slice read-only and filesystem-local. A new `internal/impact` package should own project walking, prefab GUID extraction, scene/prefab reference scanning, nested-prefab depth tracking, and deterministic result shaping. `internal/app` and `internal/cli` should stay thin: validate arguments, call the impact scanner, and format stable envelopes. This foundation intentionally stops before `prefab set impact-first` and before any suggestion heuristics.

**Tech Stack:** Go 1.22+, standard library only, existing `internal/parser`, `internal/app`, `internal/cli`, `encoding/json`, `os`, `path/filepath`, `sort`, `strings`, `testing`

---

## Scope Decision

This plan intentionally narrows `v0.5` to the smallest coherent read-only subsystem:

- include now: `prefab impact`, project-root scanning under `Assets/`, deterministic scene/prefab hit listing, nested prefab depth warning, compact output, JSON output, fixture coverage, docs
- exclude now: `prefab set`, `--allow-name-fallback`, asset impact, index acceleration, test recommendations in output, `suggest near/grid/floor`

Why this slice:

- `prefab set impact-first` depends on a trustworthy impact scanner
- `suggest` can wait; it is a different subsystem with heavier semantics
- this gives agents a safe “blast radius” tool before more writes are introduced

## Command Contract For This Slice

New command:

```bash
unity-ctx prefab impact Assets/Prefabs/Enemy.prefab --project /Users/me/MyUnityProject
unity-ctx prefab impact Assets/Prefabs/Enemy.prefab --project /Users/me/MyUnityProject --scenes Assets/Scenes/Stage01.unity,Assets/Scenes/BossRoom.unity --json
```

Rules:

- `impact` is implemented only for the `prefab` namespace in this slice.
- `<file>` is the target prefab file on disk.
- `impact` requires `--project`.
- the target prefab file must resolve under the provided Unity project `Assets/` tree.
- the target prefab must contain exactly one GUID-bearing prefab asset identity source for this slice:
  - use sidecar `.meta` first when available
  - if `.meta` is missing, report `ERROR prefab meta not found ...`
- scan scope defaults to the project `Assets/` tree for `.unity` and `.prefab` files.
- `--scenes` is optional and comma-separated; when present it limits scene scanning to those `Assets/...` scene files but still scans prefabs globally.
- nested prefab traversal depth limit is `3`; going deeper must emit a `WARN IMPACT_DEPTH_LIMIT ...`.
- output ordering must be deterministic:
  - scenes sorted by path
  - prefabs sorted by path
  - hits within a file sorted by ascending `fileID`

Expected compact output:

## Foundation Semantics

Use the simplest deterministic reference model first:

1. Resolve target prefab asset path as `Assets/...`.
2. Read `<prefab>.meta` and extract the prefab GUID.
3. Walk candidate `.unity` and `.prefab` files under `Assets/`.
4. Parse files with the existing YAML parser.
5. Match references by structured inline-map `guid` values when available.
6. Count a hit when any block contains a matching `guid` field inside an inline map or nested map.
7. Record hit `fileID` from the block header that contained the matching map.
8. For `.prefab` files, compute nested depth by following blocks that reference the target GUID and then counting repeated nesting layers inside the same file up to depth `3`.
9. If depth `3` is reached and more matching nested layers remain, set `depth_limit_hit=true` and return `WARN`.

Non-goals for this slice:

- no semantic classification of “direct instance” vs “renderer mesh dependency”
- no index reuse
- no test recommendation output
- no auto-rewrite or impact gating of writes
