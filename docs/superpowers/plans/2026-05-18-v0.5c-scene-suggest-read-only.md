# v0.5c Scene Suggest Read-only Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a read-only `unity-ctx scene suggest` command that proposes deterministic placement candidates near an anchor object using existing bounds manifests and prefab bounds, without mutating scene files.

**Architecture:** Keep `suggest` as a pure planning layer on top of the existing `bounds`, `check`, and `patch` foundations. A new `internal/suggest` package should own anchor resolution, candidate generation, floor/grid alignment, candidate scoring, and deterministic result shaping. `internal/app` and `internal/cli` should stay thin: validate command arguments, call the planner, and format stable compact/JSON envelopes. `suggest` must not write files or create patches in this slice.

**Tech Stack:** Go standard library, existing `internal/bounds`, `internal/check`, `internal/document`, `internal/parser`, and the current CLI/service test harness.

---

## Scope Check

This plan intentionally narrows `v0.5c` to the smallest coherent `suggest` slice:

- include now: `scene suggest`, `--near`, `--count`, `--align grid|floor`, compact output, JSON output, deterministic candidate ordering, fixture coverage, docs
- exclude now: `--align wall`, patch/apply execution, interactive approval, rules inference, scene writes, index acceleration

This is the right split because:

- `scene patch` and `scene apply` already exist and should remain the write path
- `scene suggest` can be valuable as a read-only planner even before deeper heuristics land
- `--align wall` is explicitly deferred in the SRS due to unclear semantics

## Command Contract

### Command shape

```bash
unity-ctx scene suggest Stage01.unity --manifest Stage01.bounds.json --prefab Assets/Prefabs/Chair.prefab --near Table_01
unity-ctx scene suggest Stage01.unity --manifest Stage01.bounds.json --prefab Assets/Prefabs/Chair.prefab --near 1000 --align floor --count 3
unity-ctx scene suggest Stage01.unity --manifest Stage01.bounds.json --prefab Assets/Prefabs/Chair.prefab --near Table_01 --align grid --json
```

### Required flags

- `--manifest`
- `--prefab`
- `--near`

### Optional flags

- `--count` (default `4`)
- `--align` (`floor` default, `grid` optional)
- `--json`

### Explicit non-goals in this slice

- no `--position`
- no `--op`
- no `--write`
- no `--patch`
- no `--align wall`
- no name fallback for mutation
- no automatic patch/apply chaining

### Selector and alignment rules

- `suggest` is implemented only for the `scene` namespace.
- `suggest` supports only compact output.
- `--near` accepts either:
  - a non-zero decimal fileID string, or
  - an exact object name from the manifest
- if a name matches multiple manifest objects, return `ERROR AMBIGUOUS_NAME name="<value>" matches=<n>`
- `--count` must be `>= 1`
- `--align` defaults to `floor`
- `--align grid` means the candidate X/Z position is snapped to a fixed `0.5` meter world grid after the near-position is computed
- `--align floor` means the candidate Y position is chosen so the prefab AABB bottom rests on world Y=`0`

### Candidate generation rules

- derive the anchor object AABB from the manifest
- derive the target prefab AABB from the manifest prefab entry
- generate exactly four candidates around the anchor in this direction order:
  1. east
  2. west
  3. north
  4. south
- edge-touch placement uses combined half extents with zero padding:
  - east/west offset from anchor center by `(anchor.size.x + prefab.size.x) / 2`
  - north/south offset from anchor center by `(anchor.size.z + prefab.size.z) / 2`
- placement coordinates are scene positions, so convert desired AABB center to command position by subtracting the prefab bounds center
- run each candidate through existing `check.CheckPlacement(...)`
- candidate status is:
  - `OK` when `check.Clear == true`
  - `WARN` when overlaps exist
- no candidate-level `ERROR` unless request validation or manifest lookup fails

### Deterministic ranking

- overall command status:
  - `OK` if at least one candidate is `OK`
  - `WARN` if all generated candidates are `WARN`
  - `ERROR` only for invalid input / missing data
- candidate sort:
  1. status rank (`OK` before `WARN`)
  2. direction order (`east`, `west`, `north`, `south`)
  3. position tuple (`x`, then `y`, then `z`) as a stable tie-breaker
- emit at most `min(count, 4)` candidates

### Compact output shape

```text
OK manifest=Stage01.bounds.json prefab=Assets/Prefabs/Chair.prefab near=Table_01 align=floor count=3 candidates=3 clear=2 warn=1
CANDIDATE rank=1 direction=east position=1.4,0,0 status=OK overlap_ids=none anchor_id=1000 anchor_name=Table_01
CANDIDATE rank=2 direction=west position=-1.4,0,0 status=OK overlap_ids=none anchor_id=1000 anchor_name=Table_01
CANDIDATE rank=3 direction=south position=0,0,-1.0 status=WARN overlap_ids=2000 anchor_id=1000 anchor_name=Table_01
```

### JSON payload shape

```json
{
  "namespace": "scene",
  "command": "suggest",
  "file": "Stage01.unity",
  "view": "compact",
  "status": "OK",
  "body": "OK manifest=... ",
  "suggest": {
    "status": "OK",
    "manifest": "Stage01.bounds.json",
    "prefab_path": "Assets/Prefabs/Chair.prefab",
    "near": {
      "file_id": 1000,
      "name": "Table_01"
    },
    "align": "floor",
    "count": 3,
    "candidates": [
      {
        "rank": 1,
        "direction": "east",
        "position": [1.4, 0.0, 0.0],
        "status": "OK",
        "overlap_ids": []
      }
    ]
  }
}
```
