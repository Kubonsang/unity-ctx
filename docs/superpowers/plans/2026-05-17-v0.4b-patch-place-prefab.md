# v0.4b Patch Place Prefab Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a read-only `scene patch place_prefab` slice that produces a deterministic prefab-placement patch plan, with compact summary by default and a structured JSON patch plan under `--json`, while deliberately deferring `apply` and `diff`.

**Architecture:** Build this slice in three layers. First, add a pure-Go scene patch planner that consumes a parsed scene, a validated bounds manifest, a prefab path, and an explicit position. Second, expose that planner through `internal/app` with stable `OK/WARN/ERROR/UNKNOWN` results and a deterministic JSON patch-plan envelope. Third, wire a narrow CLI contract for `scene patch place_prefab` without introducing any scene writes, backups, or YAML rewrites.

**Tech Stack:** Go 1.22+, standard library only, existing `internal/parser`, `internal/document`, `internal/bounds`, `internal/check`, `internal/app`, `internal/cli`, `testing`, `encoding/json`, `math`, `os`, `path/filepath`, `strconv`, `strings`

---

## Scope Decision

This plan intentionally keeps `v0.4b` narrow:

- include now: `scene patch place_prefab`, deterministic patch plan schema, compact summary output, `--json` structured plan output, fixture-backed planning tests, docs
- exclude now: `scene apply`, `scene diff`, actual `.unity` mutation, `.bak`, GUID allocation from Unity Editor, prefab metadata lookup through `.meta`, rule inference, rotation policy beyond explicit zero/default handling

Why this slice:

- `v0.4a` already locked the read-side geometry contract with `scene check`.
- the next useful step for vibe-coding agents is “tell me the exact intended patch plan,” not “write the scene immediately.”
- separating plan generation from scene mutation keeps the contract inspectable, testable, and token-efficient.

## Proposed Command Contract For This Slice

New command in this slice:

```bash
unity-ctx scene patch testdata/scenes/simple_scene.unity \
  --op place_prefab \
  --manifest testdata/manifests/simple_scene.bounds.json \
  --prefab Assets/Prefabs/chair.prefab \
  --position 5,0,0
```

JSON form for agents:

```bash
unity-ctx scene patch testdata/scenes/simple_scene.unity \
  --op place_prefab \
  --manifest testdata/manifests/simple_scene.bounds.json \
  --prefab Assets/Prefabs/chair.prefab \
  --position 5,0,0 \
  --json
```

Compact output examples:

```text
OK op=place_prefab prefab=Assets/Prefabs/chair.prefab position=5,0,0 create_objects=2 warnings=none
WARN op=place_prefab prefab=Assets/Prefabs/chair.prefab position=0,0,0 create_objects=2 warnings=AABB_OVERLAP ids=1000
UNKNOWN op=place_prefab prefab=Assets/Prefabs/chair.prefab position=5,0,0 reason=NEED_PREFAB_GUID
ERROR patch requires --op
ERROR patch requires --op place_prefab
```

Rules:

- `patch` in this slice is implemented only for the `scene` namespace.
- `patch` in this slice supports only `--op place_prefab`.
- default output is compact, stable, single-line summary.
- `--json` returns the normal result envelope with `body` equal to the compact summary plus a deterministic `patch_plan` object.
- `patch` does not write files and does not accept `--write`.
- `patch` must reject any scene/manifest mismatch, non-finite position values, and unsupported view modes.
- `WARN` may be returned for overlap findings while still producing a patch plan.
- `UNKNOWN` may be returned for missing prefab GUID or unresolved prefab-reference materialization while still giving a partial plan only if the missing parts are explicit and deterministic.

## Patch Plan Contract

The planner should emit a structured plan that future `apply` can consume without changing the contract.

Suggested JSON shape:

```json
{
  "status": "OK",
  "namespace": "scene",
  "command": "patch",
  "file": "testdata/scenes/simple_scene.unity",
  "view": "compact",
  "body": "OK op=place_prefab prefab=Assets/Prefabs/chair.prefab position=5,0,0 create_objects=2 warnings=none",
  "patch_plan": {
    "version": 1,
    "op": "place_prefab",
    "scene": "testdata/scenes/simple_scene.unity",
    "prefab": {
      "path": "Assets/Prefabs/chair.prefab",
      "guid": "UNKNOWN"
    },
    "position": [5.0, 0.0, 0.0],
    "warnings": [],
    "checks": {
      "clear": true,
      "overlap_ids": []
    },
    "planned_file_ids": {
      "game_object": 3000,
      "transform": 3001
    },
    "ops": [
      {
        "kind": "append_object",
        "class_id": 1,
        "file_id": 3000
      },
      {
        "kind": "append_object",
        "class_id": 4,
        "file_id": 3001
      }
    ]
  }
}
```

Patch-plan rules for this slice:

- `version == 1`
- `op == "place_prefab"`
- deterministic key ordering through normal `encoding/json`
- deterministic fileID planning derived from current max scene fileID
- no YAML text diff payload yet
- `ops` describes intent only: append new object blocks and the minimal fields they must carry
- missing prefab GUID should be explicit as `UNKNOWN`, not guessed

## Planner Boundary

The planner in `internal/patch` should not rewrite YAML. It should answer only:

- is this a valid `place_prefab` request?
- does the manifest correspond to the scene?
- what warnings or unknowns apply?
- what new fileIDs would be reserved?
- what object blocks would future `apply` need to materialize?

Keep prefab insertion intentionally minimal in `v0.4b`:

- plan one `GameObject` block
- plan one `Transform` block
- optionally include placeholder prefab-instance/reference info only when it can be derived deterministically without guessing

If a full prefab-instance materialization cannot be derived safely from current local data, return `UNKNOWN NEED_PREFAB_GUID` or `UNKNOWN NEED_PREFAB_RULE` with a partial but explicit plan. Do not invent GUIDs or hidden structure.

}
